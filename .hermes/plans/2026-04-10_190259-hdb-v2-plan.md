# hdb v2 — Implementation Plan

> *hdb v2 er ét CLI til at køre, overvåge, styre og interagere med hele Hermes-økosystemet — fra systemd-services til agent-sessions til miljøkonfiguration. Fra infrastruktur til interaktion.*

## Vision

hdb i dag er et operations-CLI (21 kommandoer, 7 systemd services, 88 tests). v2 tilføjer to nye lag ovenpå det eksisterende fundament:

1. **Agent-laget** — synlighed og kontrol over agenten, IKKE chat
2. **Platform-laget** — backup, security audit, miljøhåndtering

Designprincip: **Operations-first.** Hver ny kommando spørger: "Er dette operations?" Hvis ikke, skip.

### Lag 2: Agent (Nyt)

Giver synlighed og kontrol over agenten. Read + control, ikke interact.

#### `hdb agent status`

**Formål:** Vis agentens nuværende tilstand — kører den? Hvilken session? Hvor mange tokens brugt i dag?

```
$ hdb agent status
  ● Agent: RUNNING
  Session: telegram:14162 (home)
  Uptime: 2h 14m
  Tokens i dag: 124,500
  Seneste handling: 3m siden (committed fix to dashboard)
  Godkendelser i kø: 2
```

**JSON:**
```json
{
  "running": true,
  "session_id": "telegram:14162",
  "session_label": "home",
  "uptime_seconds": 8040,
  "tokens_today": 124500,
  "last_action": {
    "type": "git_commit",
    "description": "committed fix to dashboard",
    "ago_seconds": 180
  },
  "pending_approvals": 2
}
```

#### `hdb agent sessions`

**Formål:** Liste alle aktive/kendte agent sessions. Se hvilke der er levende, hvornår de startede, hvad de arbejder på.

```
$ hdb agent sessions
  ● telegram:14162 (home)     2h 14m  "hdb v2 plan"
  ○ cron:rendetalje-weekly    12m    "weekly resume — completed"
  ○ api:mcp-bridge           idle    last active 3h ago
```

**Subcommands:**
- `hdb agent sessions` (default: list active)
- `hdb agent sessions --all` (include dead/cached)

**JSON:**
```json
{
  "sessions": [
    {
      "id": "telegram:14162",
      "label": "home",
      "status": "active",
      "uptime_seconds": 8040,
      "current_task": "hdb v2 plan",
      "started_at": "2026-04-10T16:48:00Z"
    }
  ],
  "total_active": 1,
  "total_cached": 2
}
```

#### `hdb agent approve [id]`

**Formål:** Godkend eller afvis en pending handling fra agenten. Samme funktionalitet som Godkendelser-siden i dashboard, men fra terminalen.

```
$ hdb agent approve
  Pending:
  #42  Restart hermes-gateway.service?  (from telegram:14162)
  #43  Commit 3 files to main?          (from cron:rendetalje)

$ hdb agent approve 42
  ✔ #42 approved — gateway restarting

$ hdb agent approve 43 --reject "wait for review"
  ✔ #43 rejected — "wait for review" sent to agent
```

**Subcommands:**
- `hdb agent approve` (list pending)
- `hdb agent approve <id>` (approve)
- `hdb agent approve <id> --reject "reason"` (reject)
- `hdb agent approve --all` (approve all — with confirmation)

**JSON:**
```json
{
  "pending": [
    {
      "id": 42,
      "action": "Restart hermes-gateway.service",
      "source": "telegram:14162",
      "requested_at": "2026-04-10T18:55:00Z"
    }
  ]
}
```

#### `hdb agent skills`

**Formål:** Se hvilke skills agenten har, hvilke der er aktive, hvornår de sidst blev brugt. Aktiver/deaktiver.

**Subcommands:**
- `hdb agent skills list` (all installed)
- `hdb agent skills enable <name>`
- `hdb agent skills disable <name>`
- `hdb agent skills info <name>` (detailed info about one skill)

**JSON (list):**
```json
{
  "skills": [
    {
      "name": "hermes-dashboard",
      "enabled": true,
      "last_used": "2026-04-10T17:00:00Z",
      "category": "hermes-dashboard"
    }
  ],
  "total": 64,
  "enabled": 58
}
```

#### `hdb agent send <message>`

**Formål:** Send en besked til en specifik agent session. Ikke "chat" — det er at sende en instruktion til et system.

```
$ hdb agent send "tjek om tunnelen er oppe" --session telegram:14162
  ✔ Message sent to session telegram:14162
```

**NOT:** Dette er IKKE `hdb agent chat`. Det hedder `send` for at signalere at det er en engangs-instruktion, ikke en samtale.

#### `hdb agent history`

**Formål:** Se seneste agent-handlinger — ikke logfiler, men struktureret historik.

```
$ hdb agent history --limit 10
  18:42  committed fix to dashboard (3 files)
  18:38  started health check
  18:15  approved session update
  17:55  restarted tunnel service
  ...
```

**JSON:**
```json
{
  "actions": [
    {
      "time": "2026-04-10T18:42:00Z",
      "type": "git_commit",
      "description": "committed fix to dashboard",
      "details": {"files": 3, "hash": "2bd7e96"}
    }
  ],
  "limit": 10
}
```

### Lag 3: Platform (Nyt)

Platform-laget handler om at holde hele systemet sikkert, backuppet og konfigureret.

#### `hdb backup`

**Formål:** Tag, list og restore backups af hele Hermes-økosystemet. Ikke bare config — alt: ~/.hermes, logs, PID files, systemd services.

Dette er den **højeste prioritet** i v2. Hvis ~/.hermes ryger, er alt væk.

**Subcommands:**
- `hdb backup create [--label "beskrivelse"]` — tar.gz af ~/.hermes + metadata
- `hdb backup list` — vis eksisterende backups med tid, størrelse, label
- `hdb backup restore <id>` — restore fra backup (med confirmation + --force)
- `hdb backup verify <id>` — verificer backup integrity
- `hdb backup prune [--keep N]` — slet gamle backups (behold de seneste N)

**Lokation:** `~/.hermes/backups/`

**JSON (create):**
```json
{
  "id": "backup-2026-04-10T19:00:00",
  "label": "pre-v2-upgrade",
  "size_bytes": 15728640,
  "files": 1247,
  "created_at": "2026-04-10T19:00:00Z",
  "services_snapshot": {
    "api": "running",
    "web": "running",
    "tunnel": "running"
  },
  "checksum": "sha256:a1b2c3..."
}
```

**JSON (list):**
```json
{
  "backups": [
    {
      "id": "backup-2026-04-10T19:00:00",
      "label": "pre-v2-upgrade",
      "size_bytes": 15728640,
      "created_at": "2026-04-10T19:00:00Z",
      "valid": true
    }
  ],
  "total": 5,
  "total_size_bytes": 78643200
}
```

**Sikkerhed:**
- `restore` kræver confirmation (eller `--force`)
- `restore` stopper services FØR restore, starter EFTER
- Backup inkluderer checksum verification
- `prune` kræver `--keep N` — aldrig auto-slet uden explicit tal

#### `hdb security`

**Formål:** Sikkerhedsaudit af Hermes-økosystemet. Ikke en SAST-scanner — en konfigurations- og permissions-checker.

**Subcommands:**
- `hdb security audit` — checker file permissions, .env exposure, open ports, SSH keys
- `hdb security rotate` — roterer API keys, tokens, genopretter .env
- `hdb security check-env` — verificerer at sensitive vars ikke er committed til git

**JSON (audit):**
```json
{
  "passed": false,
  "findings": [
    {
      "severity": "high",
      "check": "env_file_permissions",
      "message": "~/.hermes/.env is world-readable (0644), should be 0600",
      "fix": "chmod 600 ~/.hermes/.env"
    },
    {
      "severity": "medium",
      "check": "open_port_external",
      "message": "Port 5174 is bound to 0.0.0.0 (external access)",
      "fix": "Bind to 127.0.0.1 in api/server.js"
    }
  ],
  "checks_run": 12,
  "passed_count": 10,
  "failed_count": 2
}
```

**NOT:** Dette er ikke `hdb security scan` — det lyder som en penetrationstest. `audit` er det rigtige ord: det checker konfiguration, ikke sårbarheder.

#### `hdb config`

**Formål:** Central styring af hdb's egen konfiguration. Ikke projekt-konfiguration — hdb's indstillinger.

**Subcommands:**
- `hdb config show` — vis alle hdb-indstillinger
- `hdb config set <key> <value>` — sæt en indstilling
- `hdb config get <key>` — hent en værdi
- `hdb config reset` — nulstil til defaults

**Konfigurationsfil:** `~/.hermes/hdb.config.json`

**Mulige indstillinger:**
```json
{
  "default_env": "development",
  "confirm_destructive": true,
  "log_rotation_threshold_mb": 1,
  "json_indent": 2,
  "monitor_interval_ms": 2000,
  "backup_keep_count": 10,
  "auto_backup_before_deploy": true
}
```

Små ting der gør CLI'en professionel.

#### `hdb audit`

**Formål:** One-shot fuld systemaudit — kombinerer health, doctor, security, env validation i én kommando.

```
$ hdb audit
  Services:   5/5 running        ✔
  Ports:      4/4 open           ✔
  Health:     5/5 endpoints OK   ✔
  Dependencies: 7/7 found       ✔
  Env:        2/2 required       ✔
  Security:   10/12 passed       ⚠ (2 findings)
  Backups:    3 available (oldest: 2d)   ✔
  
  Overall: HEALTHY (warnings present)
```

**JSON:**
```json
{
  "overall": "healthy",
  "warnings": 2,
  "services": {"running": 5, "total": 5},
  "ports": {"open": 4, "total": 4},
  "health": {"ok": 5, "total": 5},
  "dependencies": {"found": 7, "total": 7},
  "env": {"required_present": 2, "required_total": 2},
  "security": {"passed": 10, "total": 12, "findings": [...]},
  "backups": {"count": 3, "oldest_age_hours": 48}
}
```

#### `--env <name>` flag

**Formål:** Kør kommandoer mod et specifikt miljø. Simpelt — ikke en remote-control plane.

```
$ hdb --env production status
$ hdb --env staging deploy
$ hdb --env development restart
```

**Implementation:**
- Læser miljø-specifik config fra `~/.hermes/envs/<name>/`
- Override af porte, service-navne, .env-sti
- Default: `development` (som i dag)

**Miljøfil:** `~/.hermes/envs/production/hdb.env.json`
```json
{
  "api_port": 5174,
  "web_port": 5175,
  "proxy_port": 5176,
  "env_file": "/opt/hermes/.env",
  "dashboard_root": "/opt/hermes/dashboard"
}
```

#### `hdb --version --verbose`

**Formål:** Udvidet versionsinfo til debugging.

```
$ hdb --version --verbose
  hdb v2.0.0
  node v20.11.0
  platform linux x64
  config: ~/.hermes/hdb.config.json
  env: development
  dashboard: v1.2.0 (2bd7e96)
  services: 5 running, 2 stopped
```

## Implementation Order

### Phase 1: Backup (højeste prioritet)
```
hdb backup create|list|restore|verify|prune
```
**Hvorfor først:** Data protection. Hvis noget går galt under v2-udvikling, kan vi restore.

**Estimat:** 2-3 timer

### Phase 2: Agent Status + Approve
```
hdb agent status|sessions|approve
```
**Hvorfor:** Højeste værdi for daglig brug. Se hvad agenten laver, godkend handlinger.

**Estimat:** 3-4 timer

### Phase 3: Config + --env
```
hdb config show|set|get|reset
hdb --env <name>
```
**Hvorfor:** Grundlag for multi-miljø. Alt efterfølgende arbejde bygger på dette.

**Estimat:** 2 timer

### Phase 4: Security Audit
```
hdb security audit|rotate|check-env
```
**Estimat:** 2 timer

### Phase 5: Agent Skills + Send + History
```
hdb agent skills|send|history
```
**Estimat:** 3 timer

### Phase 6: Audit + --version --verbose
```
hdb audit
hdb --version --verbose
```
**Estimat:** 1-2 timer

## Design Constraints (Selv-imponerede regler)

1. **Operations-first.** Hver kommando spørger: "Er dette operations?" Hvis ikke, skip.
2. **Agent-laget er read + control, ikke interact.** Intet `hdb agent chat`. Kun `send`.
3. **Platform-laget er Hermes-fokus, ikke cloud-platform.** Ingen remote-control, ingen multi-server.
4. **Alle nye commands har --json** med stable schemas.
5. **POSIX exit codes** på alt (0/1/2).
6. **Confirmation + --force** på alle destruktive handlinger.
7. **Graceful degradation.** Hvis agenten er nede, vis cached data med warning.
8. **Zero breaking changes.** v1 → v2 er opt-in, ikke migration.

## Risici

| Risiko | Mitigering |
|--------|-----------|
| Agent API (5174) har ingen auth-endpoint | Tilføj auth check i agent-client.js — brug DASHBOARD_TOKEN |
| Backup af store ~/.hermes kan tage tid | Progress spinner + streaming tar |
| `--env` flag kan forvirre hvis miljø ikke findes | Check env eksisterer FØR execution, exit 2 med klar besked |
| Security audit kan give falsk positiver | Alle findings har severity + fix-forslag, ikke bare alarm |
| Scope creep i platform-laget | Fast regel: ingen remote-control, ingen multi-server i v2 |

## Åbne spørgsmål

3. **Config format:** JSON til hdb.config.json? Eller TOML? (Foreslår JSON — ingen ekstra parser)
4. **Agent-client auth:** Hvordan autentificerer hdb sig mod agent-API? Token fra ~/.hermes/.env?
